# 0.1 Daemon Lifecycle Design

This note documents the `0.1.0` daemon lifecycle implementation and how it should evolve. It is intentionally narrower than the full bridge runtime.

## Scope

The first daemon milestone provides local lifecycle commands:

- `ferris-agent-bridge start`
- `ferris-agent-bridge run`
- `ferris-agent-bridge status`
- `ferris-agent-bridge stop`

This milestone does not include IM adapters, Lark / Feishu channel integration, OpenAPI calls, real agent execution, session continuity, or streaming replies.

The goal is to prove that the project can manage one local background process safely before any remote chat event can trigger local work.

## Support Policy

`0.1.0` supports macOS as the first validated target.

CI policy for this milestone:

- macOS is the required gate and runs formatting, Clippy, locked build, and locked tests.
- Linux and Windows are build-only compatibility gates. They run locked builds to catch obvious compile regressions, but daemon lifecycle behavior is not a supported contract yet.

## Current Model

The CLI layer parses commands and delegates daemon behavior to the daemon module. The daemon module owns local runtime paths, status inspection, lock handling, process startup, stop requests, and cleanup.

The current lifecycle states are:

```text
Starting
Running
RunningUnverified
Stopped
Stale
Unowned
```

These are daemon process states, not the future runtime or run state machine. The runtime state machine planned for `0.2.0` should model sessions, queued events, run lifecycle, recovery, acknowledgements, and outbound delivery.

## Runtime Directory

The runtime directory defaults to:

```text
~/.ferris-agent-bridge
```

It can be overridden with:

```text
FERRIS_AGENT_BRIDGE_HOME
```

The current files are:

- `daemon.lock`: startup and ownership lock.
- `daemon.state`: recorded daemon metadata.
- `daemon.stop`: stop request containing the daemon token.
- `daemon.log`: local daemon log output.
- `daemon.lifecycle.lock`: transient lifecycle operation lock.

`daemon.lock` is acquired with create-new semantics so concurrent starts cannot both become owners. Once the daemon process is ready, ownership metadata is replaced atomically. Invalid lock contents are treated as unsafe and are not deleted automatically.

`daemon.state` records the process id, ownership token, executable path, start timestamp, and mode. It is written with atomic replace semantics so `status` and `stop` do not observe half-written state. The ownership token is used so `stop` does not trust a PID alone.

On Unix and macOS, the runtime directory is restricted to `0700` and daemon files are restricted to `0600`. Windows-specific ACL hardening should be finalized when the OS service adapter layer is introduced.

## Command Behavior

`start` checks the current status, removes stale state, acquires the lifecycle lock, writes a `starting` lock, spawns an internal daemon process, detaches the background child from the caller's session where supported, releases the lifecycle lock, and waits briefly for a ready record with the expected child PID and ownership token. The internal daemon entrypoint must acquire the lifecycle lock and validate that the current lock is a matching `starting` lock for the same token and starter PID before it can replace `daemon.lock` or write `daemon.state`. Startup readiness must not depend on external process command-line inspection.

`run` uses the same state and stop-request path but keeps the daemon loop in the foreground for development and debugging.

`status` reads the recorded state and validates that the process is still alive. If command-line identity evidence is available, it verifies the daemon token and internal entrypoint. If command-line identity evidence is unavailable, status may report the daemon as running but unverified.

`stop` returns not running without creating the runtime directory when no runtime directory exists. If lifecycle files exist, it acquires the lifecycle lock, writes a token-protected stop request, and waits for graceful exit. The daemon process exits when it observes the stop request; the `stop` command owns lifecycle file cleanup while holding the lifecycle lock. It only falls back to process termination after verified ownership. Before sending a force-stop signal, it must re-read state and re-check process identity so a stale PID cannot be terminated after PID reuse. It refuses to force-stop an unowned or unverified process.

Cleanup is ownership-conditional. A stop or failed-start path should only remove runtime files whose metadata still matches the daemon record or token it is cleaning up. Lifecycle operations are serialized by `daemon.lifecycle.lock`, so matching and deletion happen in the same lifecycle critical section. This prevents an old cleanup path from deleting files already written by a newer daemon start without introducing more granular file-lock protocols.

The lifecycle lock wait timeout should be longer than the slowest expected lifecycle operation. It currently allows enough room for graceful stop plus verified force-stop fallback.

Repeated starts should not create duplicate active daemon processes.

## Safety Rules

The daemon foundation should preserve these rules as it evolves:

- Do not trust PID alone.
- Treat unreadable or invalid state as stale or unsafe, not as running.
- Refuse to stop a process when ownership cannot be validated.
- Treat missing command-line identity evidence as unverified, not trusted.
- Validate the internal daemon entrypoint against the startup lock before writing runtime state.
- Do not make startup readiness depend on external command-line inspection.
- Re-check state and process identity before force-stopping a daemon.
- Keep startup lock acquisition atomic.
- Keep lock and state metadata updates atomic.
- Serialize lifecycle operations that mutate daemon lifecycle files.
- Keep runtime directory and daemon files private to the current user.
- Keep cleanup ownership-conditional so stale cleanup cannot remove a newer daemon's files.
- Keep cleanup best-effort but deterministic.
- Keep daemon lifecycle independent of any IM platform or agent CLI.

## Relationship to OS Services

The current implementation starts a child process directly. This is enough for the first local lifecycle milestone, but it is not a full OS service manager.

Future evolution should split two concepts:

- `run` or an internal daemon entrypoint: runs the bridge runtime process in the foreground.
- `start` / `stop` / `status`: manage a background process today, and a platform service registration when OS service support is enabled.

The platform service layer should be replaceable behind a small adapter boundary. Likely implementations:

- macOS: launchd LaunchAgent.
- Linux: systemd user unit.
- Windows: Task Scheduler.

The service layer should own service registration files, OS-specific status parsing, restart behavior, autostart behavior, and stdout/stderr log routing. The runtime process should still own bridge state, locks, sessions, and adapter orchestration.

## Future Multi-Agent Locking

`daemon.lock` is a process-level daemon lock. It should not become the business lock for every future agent, profile, or chat application.

The preferred long-term model is one local daemon process that can coordinate multiple profiles and agent kinds internally. Business-level exclusivity should use narrower locks, for example:

```text
registry/locks/profile/<profile>.lock
registry/locks/app/<app_id>.lock
registry/locks/agent/<agent_id>.lock
```

Profile locks prevent the same profile from running conflicting work twice. App locks prevent two runtimes from competing for the same chat platform app or event connection. Agent locks can be added later only if a specific local agent resource cannot safely run concurrently.

This keeps OS service management simple while still allowing multiple configured agents behind one daemon.

## Evolution Plan

`0.1.0` should finish local daemon lifecycle:

- Keep the current local daemon commands.
- Strengthen integration tests for duplicate starts, stale state, and stop behavior.
- Keep macOS as the first target while avoiding unnecessary platform coupling.

`0.2.0` should add the runtime foundation:

- Session and queue state.
- Run state machine.
- Inbound event ledger.
- Ack-after-persist contract.
- Outbound outbox.
- Startup recovery.

Later service-management work can add OS service adapters without changing the runtime state machine or IM adapter contracts.
