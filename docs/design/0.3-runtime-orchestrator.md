# 0.3 Runtime Orchestrator Design

## Goal

Milestone 3 layers durable orchestration on top of the runtime foundations without introducing real IM providers or real agent CLI adapters yet. The runtime should own cross-platform ordering, duplicate handling, acknowledgement preconditions, queueing, recovery, and policy decisions. Concrete providers and agents remain behind adapter boundaries.

This design note records the M3 invariants as they are introduced across focused PRs. It is not a compatibility promise during early `0.x` development.

## Boundaries

Included:

- Durable inbound event ledger for duplicate delivery handling.
- Store-level ack-after-persist persistence primitive for inbound events.
- Initial runtime intake boundary that acknowledges an inbound delivery only after durable record or duplicate detection.
- Runtime-owned event identity rules for provider/scope-safe duplicate detection.
- Process-local serialization for same-path `StateStore` read-modify-write helpers.
- Durable outbound outbox records and store-level enqueue-before-send persistence primitive.
- Outbox consumption state primitives for claiming queued deliveries and recording delivered, failed, or uncertain attempts.
- A single-step outbox worker with bounded retry scheduling and a normalized outbound adapter handoff.
- Durable per-scope inbound message queueing with debounce and bounded batching.
- Process-local scope exclusion with atomic message-batch claim, pending-run creation, durable run inputs, and queue removal.
- Store-level startup reconciliation for pending, running, interrupted, and failed runs.
- Store-level startup reconciliation and explicit resolution for ambiguous outbound deliveries.
- Policy and adapter capability skeletons for later concrete orchestration.
- Store-level durable agent-run handoff and outcome transitions.
- Later M3 concrete orchestration across the existing boundaries.

Excluded:

- Real Lark / Feishu, Slack, Discord, or other provider transports.
- Real agent process execution.
- Cross-process queueing or cross-process `StateStore` writer coordination beyond daemon ownership.
- A full platform SDK abstraction.

## Inbound Delivery Intake

Inbound transports may require an explicit delivery acknowledgement. The runtime must persist or de-duplicate the normalized event before the adapter acknowledges the platform delivery.

The initial shape is:

```text
InboundDelivery -> RuntimeOrchestrator::accept_inbound_delivery -> ImAdapter::acknowledge_inbound_delivery
```

`RuntimeOrchestrator::accept_inbound_delivery` records the event through `StateStore::persist_inbound_event`. A platform `MessageReceived` event must contain a user-authored message bound to a known session; its inbound ledger record and queue entry are written in the same atomic state replacement. When persistence returns `Recorded` or `Duplicate`, the orchestrator constructs an acknowledgement object and passes it to the adapter. If persistence fails, the adapter is not called and the platform delivery remains unacknowledged so the transport can retry according to provider semantics. If adapter acknowledgement fails, the error is reported after the event and any message queue entry are already durable.

The acknowledgement value is created by the crate, not by external callers, so adapters cannot forge an acknowledgement that skipped runtime persistence.

## Event Identity

`EventId` is the duplicate-detection key. Provider delivery identifiers are often unique only within a provider, app, tenant, chat, channel, or receive stream, so adapters must namespace inbound delivery ids before handing events to the runtime.

`EventId::for_platform_delivery(platform, scope, delivery_id)` provides the default canonical shape for this namespace. The runtime treats the resulting id as opaque after construction.

## StateStore Safety

M3 introduces read-modify-write helpers that can be reached through cloned or independently constructed `StateStore` handles. The store normalizes path aliases before choosing the I/O path, while all `StateStore` writes deliberately share one process-wide mutex. A stable per-path lock identity cannot be derived before every path exists because aliases may become resolvable later through dangling symlinks, case-insensitive filesystems, short names, or newly created ancestors. This conservative serialization preserves write ownership without changing the I/O path and prevents two in-process inbound deliveries from both being acknowledged while one ledger entry is lost to last-writer-wins replacement.

`StateStore::save` preserves inbound ledger entries and queued messages already present on disk when saving an older in-memory snapshot. Before merging, event ids present in both snapshots must agree on their compact ledger record, queue presence, and queued message identity; disagreement fails closed in either direction. Existing durable append-only records remain the ordered prefix and candidate-only records are appended, so stale writers cannot reverse already committed queue work. Candidate enqueue times are raised to the durable per-session tail when necessary, preserving monotonic queue time without changing commit order. It fails closed if the existing file cannot be read, decoded, or validated, because overwriting invalid state could otherwise erase an already acknowledged delivery or its pending work.

The same stale-save and durable-prefix ordering protection applies to outbound outbox records. A stale in-memory snapshot must not erase or move ahead of a delivery that was already durably queued by another same-path `StateStore` handle. If the same outbound delivery id appears with different payload or state, or if preserving an existing outbound delivery would require a session that the candidate state does not contain, saving fails closed instead of choosing a last-writer-wins record or replacing the existing file. Public full-state saves may introduce only new `pending` outbox records; they reject synthesized delivering, delivered, failed, or uncertain records so callers cannot bypass worker-owned transitions and retry classification.

This mutex is not a cross-process queue. The daemon lifecycle keeps one owned runtime process today; later orchestrator work should make cross-process runtime coordination explicit if the project adds more writers.

## Per-Scope Message Queue

Every newly recorded platform `MessageReceived` event produces one `QueuedMessage` in the same durable state update as its inbound ledger record. The queue entry retains the event id, scope-derived session id, normalized message, receive time, and monotonic enqueue time. A repeated event id returns `Duplicate` without adding another queue entry only after receive time and queue presence agree and, for a message, source, user authorship, session ownership, and the existing queued payload match. Malformed message duplicates, changes between message and non-message event kinds, conflicting queued-message payloads, and message event ids with no corresponding queue record fail closed. Queue records must reference both an existing inbound ledger record with the same receive time and a known session. Their global order must remain a subsequence of inbound ledger order, while their per-session enqueue times must remain monotonic, so state loading rejects missing relationships, backward timestamps, and equal-timestamp vector reordering.

`MessageQueuePolicy` applies debounce independently per session and caps each batch at `max_batch_size`. A partial batch becomes ready after no newer message for that session arrives during `debounce_secs`; a full batch becomes ready at once. When several scopes are ready, polling chooses the earliest ready time and then stable session identity, so behavior is deterministic and one noisy scope cannot extend another scope's debounce window.

`StateStore::poll_message_queue` remains a non-consuming observation API and starts no timer or worker thread. It reports either one ready batch or the next time any queued scope could become ready, without considering active runs. `StateStore::claim_message_batch` is the consuming ownership boundary: while holding the process-local write lock, it excludes every session with a pending, running, or interrupted run, reapplies the same deterministic queue policy to the remaining scopes, and atomically creates a pending `RunRecord`, persists an immutable `RunInputRecord`, and removes exactly the claimed bounded queue prefix. A terminal run releases its session for the next claim. Public snapshot saves cannot introduce run inputs and therefore cannot bypass this transition.

`RunInputRecord` retains the normalized messages after queue removal so startup recovery can classify pending, running, or interrupted ownership without losing its input history. State validation requires each input to match its run and session, retain inbound ledger identity and order, remain disjoint from the queue and every other run input, and preserve the per-session ownership sequence across historical run inputs followed by remaining queued messages. Version 4 state files still migrate to version 5 with an empty queue because their compact ledger lacks message payloads. Version 5 migrates to version 6 with empty run inputs; a version 5 file containing multiple non-terminal runs for one session fails closed rather than guessing an owner.

A successful claim is returned only after the state replacement completes. A pre-replace persistence failure leaves the queue untouched and returns no handoff. A post-replace error is durability-ambiguous: it also returns no handoff, but the pending run and its input may already own the removed messages on disk. Later startup recovery must reload that durable ownership instead of blindly creating another run. Same-process concurrent claims serialize through the state-store lock; cross-process coordination remains outside this milestone.

## Run Startup Recovery

`StateStore::reconcile_runs_at_startup` is a store-level reconciliation primitive, not agent execution or a recovered-work handoff. While holding the process-local `StateStore` write lock, it reloads durable state and classifies every recoverable run so repeated or concurrent startup calls observe one serialized state transition.

A `pending` run with a matching durable `RunInputRecord` stays pending and is reported as resumable because no agent execution has started. Recovery does not return that input as an execution handoff; the later concrete orchestrator must durably start a run before invoking an agent. A pending run without durable input cannot be reconstructed and becomes `interrupted`, making the missing ownership data explicit instead of pretending the run is resumable.

A `running` run also becomes `interrupted`. Process termination makes its agent-side effects ambiguous, so startup does not restart it or relabel it as an ordinary agent failure. `Interrupted` is a non-terminal, recovery-required state that continues to exclude its scope until a later caller explicitly resolves it as failed or cancelled; this prevents new work from overlapping a possibly surviving child process or unresolved side effect. Existing `failed` runs remain failed and are reported for inspection without automatic retry. Completed and cancelled history needs no recovery action, while an already interrupted run remains visible in the report so a retry after a durability-ambiguous write can reconstruct the same result.

The reconciliation replacement must complete before a report is returned. Pre-replace failure leaves the previous state visible and returns no report. Post-replace failure also returns no report, but a later call rebuilds the report from the durable pending, interrupted, and failed states. Recovery timestamps do not move backwards when the wall clock rolls back. Version 6 state migrates unchanged to version 7, whose run status vocabulary includes `interrupted`; historical versions reject that newer status.

The associated run input remains durable for inspection while an interrupted run continues to own its scope. The public report contains only categorized run ids, not inputs or executable handoffs. This slice does not wire recovery into the current placeholder daemon loop, execute an agent, or automatically resolve or retry a run; those remain later orchestration work.

## Outbound Outbox

Outbound replies are recorded in durable state before any future adapter is allowed to send them. `OutboundDeliveryRecord` stores the delivery id, target session, agent/system-authored message, delivery status, timestamps, attempt count, optional delivery time, and optional last error.

The initial outbox states are:

- `pending`: queued and not yet attempted.
- `delivering`: claimed for an attempted delivery.
- `delivered`: successfully handed to the future IM adapter.
- `failed`: attempted and failed, with a non-empty last error.
- `uncertain`: the adapter cannot prove whether the provider accepted the delivery, so automatic retry is unsafe.

`StateStore::enqueue_outbound_delivery` performs a read-modify-write under the same process-local `StateStore` write lock used by inbound delivery persistence. It accepts only pending records for known sessions, returns `Queued` only after the delivery is durable, returns `Duplicate` for an identical already-queued record, and fails closed for conflicting ids or failed persistence.

Crate-internal outbox consumption primitives load state under the process-local `StateStore` write lock, transition one policy-eligible pending or failed delivery to `delivering`, persist the claim, and only then return an adapter attempt. The paired adapter-outcome mark methods are also crate-internal so external consumers cannot bypass retry policy or structured adapter-failure classification. Public reconciliation resolution is a separate boundary: it accepts only an `uncertain` record plus explicit accepted or confirmed-not-accepted evidence. The unfiltered claim helper exists only in tests; `OutboxWorker` is the public production consumption boundary.

`OutboundRetryPolicy` defines a maximum total attempt count plus bounded exponential retry delays. Pending deliveries become eligible at their `updated_at_unix`; this is immediate for normally enqueued records while preserving monotonic timestamps if the wall clock moves backwards. A failed delivery is eligible at `updated_at_unix + min(initial_delay * 2^(attempts - 1), max_delay)` while its attempt count remains below the configured maximum. Exhausted deliveries remain `failed` with their last error and are not claimed again by that policy.

The crate-internal `StateStore::claim_next_outbound_delivery_attempt` applies that policy while holding the process-local `StateStore` write lock. It skips deferred and exhausted failures, persists the selected record as `delivering`, and then returns an `OutboundDeliveryAttempt` containing the stable delivery id, normalized session scope, message, and attempt number.

`OutboxWorker::process_next` processes at most one eligible delivery. It sends only after the claim is durable, then persists `delivered`, retryable `failed`, or non-retryable `uncertain` before returning an outcome. `ImAdapter` returns a structured `OutboundDeliveryFailure`: `Retryable` means the adapter can establish that the provider did not accept the delivery, while `Uncertain` means acceptance may already have happened and automatic retry could duplicate the message. Only `Retryable` outcomes participate in backoff.

A claim or outcome persistence failure is a recoverable runtime error. These writes atomically replace the state file before a final parent-directory sync, so an error may leave either the previous or updated state visible. A claim error never calls the adapter, but may leave `pending`, `failed`, or `delivering` on disk depending on whether the attempt was initial or a retry and whether replacement completed. The worker therefore reports the durable state as uncertain and requires callers or recovery code to reload it before deciding what happened. A delivered or uncertain adapter outcome is never converted into a retryable failure merely because recording it returned an error.

The worker is deliberately a deterministic single-step component rather than a self-spawned background thread. Later concrete runtime orchestration owns wakeups and repeated polling. `StateStore::reconcile_outbound_deliveries_at_startup` provides platform-neutral recovery for records left `delivering`: it preserves ambiguous attempts for explicit reconciliation instead of blindly retrying them. Real provider send APIs remain M5 work, where concrete adapters can refine reconciliation using provider idempotency keys or delivery-status queries when available.

## Outbound Startup Reconciliation

The outbox recovery boundary reuses the existing `uncertain` status as the explicit reconciliation-required state. A durable `delivering` record means the adapter handoff may already have happened, but startup cannot distinguish that case from a claim replacement that became visible before the worker called the adapter. `StateStore::reconcile_outbound_deliveries_at_startup` reloads state under the process-local write lock, moves every remaining `delivering` record to `uncertain` with a non-empty recovery diagnostic and a nondecreasing timestamp, and reports every `uncertain` delivery id. Existing `uncertain` records remain unchanged and appear in the same report so retries after durability-ambiguous writes reconstruct the complete unresolved set.

The recovery report contains ids only, not adapter attempts or retry handoffs. Pending and retryable failed records remain under normal worker policy, while delivered records require no recovery. Recovery does not increment attempt counts, call an adapter, retry a send, or guess a provider outcome.

An uncertain delivery can leave reconciliation only through an explicit store operation. `StateStore::resolve_outbound_delivery_as_delivered` is valid when a provider query or operator evidence confirms acceptance. `StateStore::resolve_outbound_delivery_as_failed` is valid only when non-acceptance is confirmed; it requires a non-empty reason and returns the record to the existing bounded retry policy without erasing its attempt history. Normal resolution rejects any source state other than `uncertain` and preserves nondecreasing timestamps. An idempotent replay may also accept an already matching target state solely to re-establish the parent-directory durability barrier after an ambiguous write. Both replay targets must match the persisted resolution timestamp, failed-state replay must also match the persisted non-empty reason, and neither replay mutates the record.

Transition and resolution writes complete before returning reports or resolved records. Pre-replace failure leaves the previous state visible. Post-replace failure returns no success, and a later recovery call or exact resolution replay rebuilds the classification or confirms the resolution from durable state only after re-establishing the parent-directory durability barrier. Missing state remains absent, repeated and same-process concurrent recovery returns one durable classification, matching same-process resolutions are idempotent, conflicting same-process resolutions have one winner, and stale public snapshot saves fail closed rather than restore an unresolved state. Cross-process `StateStore` writer coordination remains outside M3 under the single-daemon ownership boundary.

This slice does not increment the runtime state schema version because it adds no serialized field or status value: versions that already persist outbound deliveries already understand `uncertain`. Real provider status queries remain M5 adapter work, and automatic startup wiring remains part of the later concrete runtime orchestrator.

## Adapter Surface

`ImAdapter` acknowledges runtime-approved inbound deliveries and accepts normalized `OutboundDeliveryAttempt` values from the outbox worker. The attempt carries a stable delivery id so a concrete adapter can use provider idempotency support when available. The structured outbound failure contract forces adapters to distinguish a request known not to have been accepted from an ambiguous transport outcome. Provider request types and API clients do not cross this boundary.

Real provider transport details stay inside concrete IM adapters. The core runtime sees normalized events, acknowledgement tokens, and adapter results only.

`AgentAdapter` accepts one normalized `AgentRunRequest` at a time. The request is derived from a durable `RunInputRecord` and contains the stable run id, session id, ordered user messages, and a validated workspace root. The runtime constructs this value only after it owns the pending run and authorizes the exact workspace; adapters cannot replace the durable input or select a different working directory.

An adapter returns an `AgentRunOutput` containing one or more agent-authored messages for the same session, or an `AgentRunFailure`. Successful output and a definite failure both assert that all execution owned by the adapter, including its process tree, has terminated; only then may the runtime complete or fail the run and release its scope. An uncertain failure means the adapter cannot prove that termination, so orchestration must preserve non-terminal ownership for explicit recovery instead of releasing the scope or retrying automatically. Filesystem or other local side effects do not by themselves make an otherwise terminated execution uncertain.

Empty output, user/system-authored output, and output for another session are rejected while constructing the normalized output. An adapter that cannot produce valid output reports a definite failure only when it can still prove complete execution termination; otherwise it reports an uncertain failure.

This M3 capability is deliberately synchronous and completion-oriented. It defines the provider-neutral handoff needed by the concrete orchestrator without spawning a process. Process handles, streaming `AgentEvent` values, stdout/stderr parsing, timeout, cancellation, backpressure, and process-tree termination remain M4 responsibilities. Adding those mechanics must preserve the request identity, workspace ownership, and definite-versus-uncertain failure semantics defined here.

## Durable Agent Run Transitions

The concrete orchestrator must not hand a pending run to an agent until the transition to `running` is durable. The store-level start operation reloads state under the process-local write lock, requires the matching immutable `RunInputRecord`, transitions the run, persists the replacement, and only then returns the input that may be converted into an `AgentRunRequest`. Two same-process callers cannot both receive a handoff for the same run. A failed start write returns no handoff; because an error after replacement may still leave `running` visible, callers must reload or use startup recovery instead of invoking the adapter speculatively.

A successful adapter result is committed as one store transaction: every output message becomes a pending `OutboundDeliveryRecord`, and the run becomes `completed` in the same state replacement. The store derives stable delivery ids from the run id and output order instead of accepting adapter-selected ids. `RunRecord::output_delivery_ids` retains that complete ordered ownership link. The operation requires a running run, exact session ownership, non-empty agent-authored output, unique delivery ids, and nondecreasing timestamps. It cannot expose completed ownership without the corresponding outbox work or expose outbox work while leaving the run running. Replaying the exact same completion after a durability-ambiguous write confirms the persisted result even if an outbox worker has since advanced delivery status; changed messages, ordering, timestamps, or run ownership fail closed.

A definite adapter failure durably moves the running run to `failed`, releasing the scope only after termination has been established. An uncertain failure durably moves it to `interrupted`, preserving non-terminal ownership for explicit recovery. Exact same-target replay is accepted only to re-establish the durability barrier after an ambiguous write; conflicting terminal or interrupted outcomes fail closed. Public snapshot saves may carry an unchanged durable agent run or a stale ancestor that merge preserves, but cannot create `running` or `completed` state for a run with durable input or introduce or change output ownership. Callers therefore cannot create a handoff or bypass the atomic completion transition through `StateStore::save`; existing failed, cancelled, and interrupted resolution paths remain available. These primitives do not evaluate policy, choose a workspace, call an adapter, schedule queue wakeups, or run the outbox worker. Those sequencing responsibilities remain in the later concrete orchestrator.

Output ownership advances the runtime state schema from version 7 to version 8. Version 7 files migrate with empty `output_delivery_ids`; versions 1 through 7 reject non-empty output links because they did not define that relationship. Version 8 requires `output_delivery_ids` to be explicitly present on every run and validates that every linked delivery exists exactly once, belongs to the run session, contains an agent-authored message, and has a creation timestamp equal to the run completion timestamp. Historical completed runs may retain an explicitly empty link, while new agent-run completions through the store primitive require non-empty output.

## Policy Foundations

M3 policy foundations are pure, platform-neutral decision components. `PolicyDecision<R>` returns either `Allowed` or `Denied` with a typed reason so orchestration can log and test the exact policy outcome without embedding provider request types or transport behavior.

`WorkspacePolicy` is an exact allowlist of validated `WorkspaceRoot` values. A workspace root must be an absolute, non-filesystem-root path with no parent-directory component. On Windows, only disk and UNC paths, including their verbatim forms, are accepted; device namespaces and generic verbatim prefixes are rejected. Evaluation rejects invalid paths and paths not present in the allowlist; an allowed parent does not implicitly authorize descendant or prefix-colliding working directories. A validated root preserves the caller-supplied lexical `OsString` identity rather than rewriting separators, prefix kinds, or path aliases. The policy performs lexical validation only and does not touch the filesystem, resolve symbolic links, or assert that a path exists. The later configuration and process-launch boundary remains responsible for any required canonicalization and existence checks before constructing the allowlist or starting an agent.

`AccessPolicy` evaluates a provider-neutral `AccessPrincipal`, a session scope, and an `AccessAction`. Principals retain separate non-empty platform and subject namespaces so adapters do not collapse identities from different providers. The policy uses exact principal and scope allowlists, defaults to denying every request, and permits `AdministerBridge` only for an administrator set that must be a subset of the ordinary allowed principals. `InvokeAgent` remains a separate non-administrative action. Denials distinguish an unknown principal, an unknown scope, and a non-administrator attempting an administrative action.

This slice does not add policy fields to persistent config or runtime state, extract provider identities from messages, emit policy logs, or wire policy evaluation into `RuntimeOrchestrator`. Those integration points belong to later concrete orchestration and the first provider milestone, so this slice does not change the runtime state schema.

## Acceptance Notes

The inbound intake slice is complete when duplicate inbound deliveries are recorded or recognized before acknowledgement, persistence failures do not acknowledge, acknowledgement failures are reported after durability, same-process same-path store handles cannot lose acknowledged inbound records, stale saves preserve existing inbound ledger entries, and the architecture/roadmap documents state that real transports and cross-process orchestration remain out of scope.

The per-scope message queue and exclusion slices are complete when session-bound inbound messages persist their ledger and queue entries atomically before acknowledgement; duplicate events do not enqueue twice before or after claim; unknown sessions and malformed queue, run-input, or ledger relationships fail closed; debounce is isolated per scope; full batches become ready without waiting; observation is deterministic and non-consuming; a durable claim creates one pending run and recoverable input while removing only the bounded queue prefix; pending, running, and interrupted runs block another claim for their session while terminal history does not; concurrent same-process claims return at most one handoff; pre-replace failures preserve queued work; post-replace ambiguity returns no handoff but preserves recoverable ownership; stale saves cannot erase claims or restore claimed messages; public snapshots cannot synthesize run inputs; version 4 migrates to an empty version 5 queue; and version 5 migrates to empty version 6 run inputs. Run startup recovery is covered by the separate acceptance boundary below.

The run startup recovery slice is complete when pending runs with durable inputs remain pending and are reported without an execution handoff; running and input-less pending runs become non-terminal interrupted owners; interrupted ownership continues blocking its scope until explicit failed or cancelled resolution; existing failed runs remain inspectable without automatic retry; completed and cancelled history is omitted; repeated and concurrent calls return the same durable classification; clock rollback does not move run timestamps backwards; pre-replace failures preserve the old state; post-replace failures return no report but a later call reconstructs it; stale saves cannot restore running over interrupted; missing state is not created; and version 6 migrates to version 7 while historical versions reject interrupted runs.

The outbound outbox slice is complete when outbound records validate session/message ownership, state transitions reject invalid delivery shapes and backward timestamps, version 3 state files migrate with an empty outbox, version 4 state files require explicit outbound deliveries, enqueue accepts only pending records for known sessions and returns only after durable persistence, duplicate enqueue is idempotent, public snapshot saves cannot introduce non-pending deliveries, same-process same-path store handles cannot lose queued outbound records, and stale saves preserve already queued deliveries or fail closed without replacing the existing file when dependencies are missing.

The outbox consumption primitive slice is complete when a claim transitions exactly one pending or failed delivery to `delivering` before returning it, failed deliveries can be claimed again for retry, no delivery is returned when no record is claimable, same-process same-path store handles cannot claim the same delivery twice, failed claim persistence returns an error instead of a handoff, and delivered or failed outcomes are durable before the updated record is returned.

The outbox worker slice is complete when only due deliveries are handed to the adapter after a durable claim, the handoff contains normalized scope and a stable idempotency key, successful, retryable-failed, and uncertain sends are durably recorded before their outcomes return, only failures known not to have been accepted enter bounded exponential backoff, uncertain outcomes remain inspectable without being reclaimed, exhausted failures remain inspectable without being reclaimed, deferred failures do not block later eligible records, claim persistence failures never trigger an adapter call, and outcome persistence errors are treated as uncertain durability rather than triggering an unsafe immediate retry.

The outbound startup reconciliation slice is complete when every startup-visible `delivering` record becomes `uncertain` without another adapter handoff or attempt increment; all existing and recovered uncertain delivery ids are reported; pending, failed, and delivered records remain unchanged; only explicit confirmed-accepted or confirmed-not-accepted resolution can leave `uncertain`; confirmed non-acceptance returns to bounded retry policy without losing attempt history; timestamps do not move backwards; missing state is not created; pre-replace and post-replace failures return no premature success; exact same-target replay confirms durability while rejecting changed timestamps or reasons; repeated and same-process concurrent calls reconstruct one durable result; stale saves cannot restore an unresolved state; and that slice leaves the then-current runtime state schema at version 7.

The policy foundation slice is complete when decisions are typed and deterministic; an empty workspace or access policy denies by default; workspace roots reject relative, parent-containing, and filesystem-root paths, plus Windows device namespaces and generic verbatim prefixes; workspace authorization preserves lexical path identity and requires an exact allowlist match rather than separator or prefix rewriting, descendant authorization, or string-prefix matching; principals preserve platform and subject namespaces; ordinary access requires both the principal and session scope to be allowlisted; administrative access additionally requires an administrator that is already an allowed principal; malformed administrator configuration fails closed; and the slice adds no persistent config, runtime-state schema, provider identity extraction, filesystem lookup, or orchestration side effect.

The agent adapter capability slice is complete when requests preserve the durable run and session identity, expose the ordered user messages without a persistence or provider type crossing the boundary, and carry only a validated exact workspace root; outputs require at least one agent-authored message for the request session; successful output and definite failure assert complete execution-tree termination while uncertain failure means termination cannot be proven; invalid output follows the same termination-based classification; the trait performs no persistence or policy decision itself; and the slice adds no process execution, event streaming, timeout, cancellation, concrete adapter, or runtime-state schema change.

The durable agent-run transition slice is complete when an adapter-eligible input is returned only after its pending run is durably running; missing input, non-pending state, same-process contention, and failed persistence return no handoff; successful output is non-empty and agent-authored, receives stable runtime-owned delivery ids, and is linked to pending outbox records atomically with run completion; definite failure becomes terminal failed while uncertain failure becomes non-terminal interrupted; timestamps do not move backwards; exact same-target replay can confirm an ambiguous replacement but conflicting replay fails closed; stale public saves cannot undo the transitions, erase committed output, or synthesize output ownership; version 7 migrates to version 8 with empty output links while historical versions reject the new linkage; and the slice adds no policy sequencing, adapter call, queue loop, or process execution.
