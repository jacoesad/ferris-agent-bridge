# 0.2 Runtime Foundations Design

## Goal

`0.2.0` establishes the platform-neutral runtime foundation for the bridge. It defines the local config, durable state, session identity, common message and event models, structured logging, redaction, and error policy that later runtime orchestration and adapters will build on.

This milestone should not start real agent runs, consume chat events, define adapter traits, or implement queueing. Those belong to later milestones.

## Boundaries

Included:

- Versioned local config with a `secret: Option<SecretInput>` placeholder.
- Runtime state storage with atomic write semantics and Unix/macOS private permissions for newly created JSON files.
- Stable session identity derived from platform and scope.
- Platform-neutral `Message` and `Event` domain models.
- Structured log events carrying session, event, and future run context.
- Secret redaction for structured fields and inline values.
- Runtime error classification into fatal, recoverable, and user-visible errors.

Excluded:

- Run lifecycle state machine.
- Inbound event ledger and outbound outbox.
- Message queueing, debounce, or batching.
- `ImAdapter` and `AgentAdapter` traits.
- Workspace/access policy evaluation.
- Lark / Feishu, Slack, Discord, or agent-specific implementation details.

## Config

`BridgeConfig` is versioned from the beginning so future releases can migrate old configs instead of breaking local setups. The first config shape is intentionally small:

- `version`: config format version.
- `profile`: user-selected local profile name.
- `runtime`: state and log file names relative to the profile/runtime directory.
- `secret`: a placeholder for future keystore-backed values.

`SecretInput` only models references such as environment variables or files. It does not persist plaintext secrets in config. The encrypted local keystore is intentionally deferred until the chat-platform milestone where real platform credentials are introduced.

## State

`RuntimeState` stores runtime-owned state as JSON and is written through a same-directory temporary file followed by `rename`. This gives the daemon and future runtime a single durable write path and avoids partial state files after process crashes.

On Unix/macOS, newly created runtime JSON files and missing parent directories are created with private permissions at creation time. On Windows, this milestone keeps the same atomic-write API and compatibility coverage but does not yet harden ACLs explicitly; Windows-specific ACL privacy should be added before these helpers store plaintext platform credentials.

The initial state contains sessions only. Runs, inbound ledgers, outbound outboxes, and recovery records are excluded until `0.3.0`.

## Sessions

Session identity is stable for a `(platform, scope)` pair. The scope is intentionally generic: for Lark it may later be a chat id or thread id; for another platform it may be a channel, conversation, or direct-message scope.

The session layer must not know provider SDK types. Platform adapters normalize provider identifiers into the common `SessionScope` shape before handing data to the runtime.

Session timestamps are part of the continuity model: `created_at_unix` records the first time the runtime saw the session, and `updated_at_unix` must not be earlier than `created_at_unix` or move backwards during refresh.

## Messages and Events

`Message` models bridge-level content, authorship, and optional session association. `Event` models normalized runtime input/output facts such as a received message, a session start, or a runtime notice.

These are platform-neutral domain models. Provider request payloads, raw event envelopes, receive ids, card actions, and media resources remain inside platform adapters.

## Logging and Redaction

Structured log events include a level, message, and context fields. The context supports:

- `session_id`
- `event_id`
- `run_id` placeholder for later milestones
- free-form fields for adapter/runtime diagnostics

Secret-like fields such as token, secret, password, and authorization are redacted before logs are emitted. This policy applies to structured fields and inline `key=value` or `key:` patterns in messages.

## Error Policy

Runtime errors are classified by behavior:

- `fatal`: the process cannot continue safely.
- `recoverable`: the operation failed but the runtime can continue.
- `user_visible`: the failure should be returned to the caller or chat user as a structured error.

Adapters added later must map provider and agent failures into this policy instead of inventing independent failure behavior.

## Acceptance Notes

This milestone is complete when config and state can be saved/loaded through the shared atomic write path, Unix/macOS-created runtime JSON files use private permissions, sessions have stable identifiers, message/event models round-trip through JSON, log output redacts secret-like data, and the existing daemon lifecycle tests continue to pass. Explicit Windows ACL hardening is deferred.
