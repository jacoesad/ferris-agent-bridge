# 0.3 Runtime Orchestrator 设计

## 目标

Milestone 3 在 runtime foundations 之上加入 durable orchestration，但暂不引入真实 IM provider 或真实 agent CLI adapter。Runtime 应拥有跨平台的 ordering、duplicate handling、acknowledgement preconditions、queueing、recovery 和 policy decisions。具体 provider 和 agent 仍留在 adapter 边界之后。

本文档记录 M3 在多个聚焦 PR 中逐步引入的设计约束。早期 `0.x` 阶段它不是兼容性承诺。

## 边界

包含：

- 用于重复投递处理的 durable inbound event ledger。
- Inbound events 的 store-level ack-after-persist persistence primitive。
- 初始 runtime intake 边界：只有 durable record 或 duplicate detection 成功后，才 ack inbound delivery。
- Runtime 拥有的 event identity 规则，用于 provider/scope-safe duplicate detection。
- 同路径 `StateStore` read-modify-write helpers 的进程内串行化。
- Durable outbound outbox records，以及 send 前必须 enqueue 的 store-level persistence primitive。
- Outbox consumption state primitives，用于领取 queued deliveries，并记录 delivered、failed 或 uncertain attempts。
- 带有 bounded retry scheduling 和 normalized outbound adapter handoff 的 single-step outbox worker。
- Durable per-scope inbound message queueing，包含 debounce 和 bounded batching。
- 进程内 scope exclusion，包含原子 message-batch claim、pending-run creation、durable run inputs 和 queue removal。
- 对 pending、running、interrupted 和 failed runs 执行 store-level startup reconciliation。
- 对 ambiguous outbound deliveries 执行 store-level startup reconciliation 和显式 resolution。
- 面向后续 concrete orchestration 的 policy 和 adapter capability skeletons。
- Store-level durable agent-run handoff 和 outcome transitions。
- 后续 M3 跨现有边界的 concrete orchestration。

不包含：

- 真实 Lark / 飞书、Slack、Discord 或其他 provider transport。
- 真实 agent process execution。
- daemon ownership 之外的 cross-process queueing 或 cross-process `StateStore` writer coordination。
- 完整 platform SDK abstraction。

## Inbound Delivery Intake

有些 inbound transport 需要显式 delivery acknowledgement。Runtime 必须先持久化或去重归一化后的 event，再让 adapter ack 平台 delivery。

初始形态是：

```text
InboundDelivery -> RuntimeOrchestrator::accept_inbound_delivery -> ImAdapter::acknowledge_inbound_delivery
```

`RuntimeOrchestrator::accept_inbound_delivery` 通过 `StateStore::persist_inbound_event` 记录 event。平台 `MessageReceived` event 必须包含绑定到已知 session 的 user-authored message；它的 inbound ledger record 和 queue entry 会通过同一次原子 state replacement 写入。当持久化返回 `Recorded` 或 `Duplicate` 时，orchestrator 创建 acknowledgement 对象并交给 adapter。如果持久化失败，不会调用 adapter，平台 delivery 保持未 ack，让 transport 按 provider 语义重试。如果 adapter acknowledgement 失败，错误会在 event 和对应 message queue entry 已经 durable 之后返回。

Acknowledgement value 由 crate 创建，而不是由外部调用方创建，因此 adapter 不能伪造一个跳过 runtime persistence 的 acknowledgement。

## Event Identity

`EventId` 是 duplicate-detection key。Provider delivery identifier 通常只在某个 provider、app、tenant、chat、channel 或 receive stream 内唯一，所以 adapter 在把 event 交给 runtime 前，必须给 inbound delivery id 加命名空间。

`EventId::for_platform_delivery(platform, scope, delivery_id)` 提供默认 canonical shape。构造完成后，runtime 把这个 id 当作 opaque value。

## StateStore Safety

M3 引入的 read-modify-write helpers 可能通过 cloned 或独立构造的 `StateStore` handles 触发。Store 会先规范化 path aliases，再选择 I/O path，同时所有 `StateStore` write 会有意共享一把进程级 mutex。在所有路径存在之前，无法推导稳定的按路径 lock identity，因为 dangling symlinks、case-insensitive filesystems、short names 或新创建的 ancestors 都可能让 alias 在稍后才变为可解析。这种保守串行化不会改变 I/O path，同时能够维持 write ownership，避免两个进程内 inbound deliveries 都被 ack，但其中一个 ledger entry 被 last-writer-wins replacement 覆盖丢失。

`StateStore::save` 在保存较旧的内存快照时，会保留磁盘上已经存在的 inbound ledger entries 和 queued messages。合并前，两个 snapshot 中都存在的 event id 必须在 compact ledger record、queue presence 和 queued message identity 上一致；任一方向不一致都会 fail closed。已有 durable append-only records 保持为有序前缀，candidate-only records 追加在后，因此 stale writer 不能反转已经提交的 queue work。必要时 candidate enqueue time 会提升到该 session 的 durable tail，从而在不改变 commit order 的前提下保持 queue time 单调。如果现有文件不能读取、解码或验证，它会 fail closed，因为覆盖 invalid state 可能擦掉已经 ack 的 delivery 或它的 pending work。

同样的 stale-save 和 durable-prefix ordering 保护也适用于 outbound outbox records。较旧的内存快照不能擦掉另一个同路径 `StateStore` handle 已经 durably queued 的 delivery，也不能移动到它前面。如果同一个 outbound delivery id 对应不同 payload 或状态，或者保留已有 outbound delivery 需要 candidate state 中不存在的 session，保存会 fail closed，而不是采用 last-writer-wins 或替换现有文件。Public full-state save 只能引入新的 `pending` outbox record；它会拒绝伪造的 delivering、delivered、failed 或 uncertain record，避免调用方绕过 worker 拥有的 transition 和 retry classification。

这把 mutex 不是跨进程队列。当前 daemon lifecycle 保证一个被拥有的 runtime 进程；如果以后项目加入更多 writer，后续 orchestrator 工作应显式设计 cross-process runtime coordination。

## Per-Scope Message Queue

每个新记录的平台 `MessageReceived` event 都会在与 inbound ledger record 相同的 durable state update 中产生一个 `QueuedMessage`。Queue entry 保留 event id、由 scope 派生的 session id、normalized message、receive time 和单调的 enqueue time。重复 event id 只有在 receive time 和 queue presence 一致，并且消息的 source、user authorship、session ownership 和已有 queued payload 都匹配时，才返回 `Duplicate` 且不再次入队。损坏的 message duplicate、message 与 non-message event kind 之间的变化、冲突的 queued-message payload，以及没有对应 queue record 的 message event id 都会 fail closed。Queue record 必须同时引用具有相同 receive time 的已有 inbound ledger record 和已知 session。其全局顺序必须保持为 inbound ledger order 的子序列，并且每个 session 的 enqueue time 必须单调，因此 state load 会拒绝关系缺失、时间戳倒退和相同时间戳下的 vector 重排。

`MessageQueuePolicy` 对每个 session 独立应用 debounce，并把每个 batch 限制为 `max_batch_size`。Partial batch 在该 session 的最后一条消息之后经过 `debounce_secs` 才 ready；full batch 立即 ready。多个 scope 同时 ready 时，poll 先选择最早 ready time，再按稳定 session identity 决定顺序，因此行为确定，并且高流量 scope 不会延长其他 scope 的 debounce window。

`StateStore::poll_message_queue` 仍是 non-consuming observation API，也不会自行启动 timer 或 worker thread。它不考虑 active run，只返回一个 ready batch，或者任一 queued scope 下一次可能 ready 的时间。`StateStore::claim_message_batch` 是消费侧 ownership boundary：它持有进程内 write lock，排除具有 pending、running 或 interrupted run 的 session，对剩余 scope 重新应用相同的确定性 queue policy，并在同一次原子 replacement 中创建 pending `RunRecord`、持久化不可变 `RunInputRecord`，再精确移除已领取的 bounded queue prefix。Terminal run 会释放该 session 供下一次 claim。Public snapshot save 不能引入 run input，因此无法绕过这一 transition。

`RunInputRecord` 会在 queue removal 后保留 normalized messages，使 startup recovery 能在不丢失 input history 的情况下分类 pending、running 或 interrupted ownership。State validation 要求每个 input 与其 run 和 session 一致，保留 inbound ledger identity 与顺序，与 queue 及其他 run input 互斥，并维持“历史 run inputs 后接剩余 queued messages”的 per-session ownership sequence。Version 4 state file 仍会迁移为空 queue 的 version 5，因为它的 compact ledger 不包含 message payload。Version 5 会迁移为空 run inputs 的 version 6；如果 version 5 对同一 session 含有多个 non-terminal run，则 fail closed，而不是猜测 owner。

只有 state replacement 完成后，成功 claim 才会返回。Pre-replace persistence failure 会保留原 queue 且不返回 handoff。Post-replace error 的 durability 不明确：它同样不返回 handoff，但 pending run 及其 input 可能已经在磁盘上拥有被移除的消息。后续 startup recovery 必须重新加载这份 durable ownership，而不能盲目创建另一个 run。同进程并发 claim 通过 state-store lock 串行化；cross-process coordination 仍不属于本里程碑。

## Run Startup Recovery

`StateStore::reconcile_runs_at_startup` 是 store-level reconciliation primitive，而不是 agent execution 或 recovered-work handoff。它会持有进程内 `StateStore` write lock，重新加载 durable state，并对每个可恢复 run 进行分类，使重复或并发 startup 调用观察到同一个串行化 state transition。

具有匹配 durable `RunInputRecord` 的 `pending` run 会保持 pending 并报告为 resumable，因为 agent execution 尚未开始。Recovery 不会把该 input 作为 execution handoff 返回；后续 concrete orchestrator 必须先 durably start run，才能调用 agent。缺少 durable input 的 pending run 无法重建，因此会转为 `interrupted`，显式暴露 ownership data 缺失，而不是假装该 run 可以恢复。

`running` run 同样会转为 `interrupted`。进程终止后，其 agent-side effects 具有歧义，因此 startup 不会重新执行，也不会把它重新标记为普通 agent failure。`Interrupted` 是 non-terminal、需要 recovery 的状态；在后续调用方显式把它解决为 failed 或 cancelled 前，它会继续排除该 scope，避免新 work 与可能仍存活的 child process 或未解决 side effect 重叠。已有 `failed` run 保持 failed，并显式报告供检查，但不会自动 retry。Completed 和 cancelled history 不需要 recovery action；已经 interrupted 的 run 仍会出现在报告中，使 durability-ambiguous write 后的重试可以重建相同结果。

Reconciliation replacement 完成后才能返回报告。Pre-replace failure 保持之前 state 可见且不返回报告。Post-replace failure 同样不返回报告，但后续调用可以根据 durable pending、interrupted 和 failed 状态重建报告。墙上时钟回拨时，recovery timestamp 不会倒退。Version 6 state 会原样迁移到 version 7，后者的 run status vocabulary 包含 `interrupted`；历史版本会拒绝这一新状态。

Interrupted run 会继续拥有其 scope，关联 run input 也保持 durable 供检查。Public report 只包含分类后的 run ids，不包含 input 或 executable handoff。本切片不会把 recovery 接入当前 placeholder daemon loop，不会执行 agent，也不会自动 resolve 或 retry run；这些仍属于后续 orchestration 工作。

## Outbound Outbox

Outbound replies 必须先记录到 durable state，后续 adapter 才能发送。`OutboundDeliveryRecord` 保存 delivery id、目标 session、agent/system-authored message、delivery status、timestamps、attempt count、可选 delivered time 和可选 last error。

初始 outbox 状态包括：

- `pending`：已经排队，尚未尝试发送。
- `delivering`：已被领取用于一次发送尝试。
- `delivered`：已经成功交给未来的 IM adapter。
- `failed`：尝试发送失败，并带有非空 last error。
- `uncertain`：adapter 无法确认 provider 是否已经接受 delivery，因此自动重试并不安全。

`StateStore::enqueue_outbound_delivery` 使用与 inbound delivery persistence 相同的进程内 `StateStore` write lock 执行 read-modify-write。它只接受已知 session 上的 pending record，只有在 delivery 已经 durable 后才返回 `Queued`，对完全相同的已排队记录返回 `Duplicate`，对冲突 id 或持久化失败 fail closed。

Crate 内部的 outbox consumption primitives 会在进程内 `StateStore` write lock 下加载 state，把一个符合 policy 的 pending 或 failed delivery 转为 `delivering`，持久化 claim，然后才返回 adapter attempt。配套的 adapter-outcome mark methods 同样限制在 crate 内部，因此外部调用方不能绕过 retry policy 或结构化 adapter-failure classification。Public reconciliation resolution 是独立边界：它只接受 `uncertain` record 和显式的“已接受”或“确认未接受”证据。未过滤的 claim helper 只存在于测试中；`OutboxWorker` 是 public production consumption boundary。

`OutboundRetryPolicy` 定义总 attempt 次数上限和 bounded exponential retry delays。Pending delivery 会在自身的 `updated_at_unix` 到达时变为 eligible；正常入队的记录可以立即领取，同时在 wall clock 回拨时维持时间戳单调性。只要 attempt count 仍低于配置上限，failed delivery 会在 `updated_at_unix + min(initial_delay * 2^(attempts - 1), max_delay)` 到期后重新变为 eligible。达到上限的 delivery 保持 `failed` 及其 last error，并且不会再被该 policy 领取。

Crate 内部的 `StateStore::claim_next_outbound_delivery_attempt` 在持有进程内 `StateStore` write lock 时应用该 policy。它跳过尚未到期和已耗尽的 failure，把选中的 record 持久化为 `delivering`，然后返回包含稳定 delivery id、normalized session scope、message 和 attempt number 的 `OutboundDeliveryAttempt`。

`OutboxWorker::process_next` 每次最多处理一个 eligible delivery。它只会在 claim durable 后发送，并且在返回 outcome 前持久化 `delivered`、可重试的 `failed` 或不可重试的 `uncertain`。`ImAdapter` 返回结构化 `OutboundDeliveryFailure`：`Retryable` 表示 adapter 可以确认 provider 未接受 delivery，`Uncertain` 表示 delivery 可能已经被接受，自动重试可能导致重复消息。只有 `Retryable` outcome 会进入 backoff。

Claim 或 outcome 持久化失败属于 recoverable runtime error。这些 write 会先原子替换 state file，之后再执行可能失败的 parent-directory sync，因此错误返回时旧状态或新状态都可能可见。Claim error 绝不会调用 adapter，但根据这是初次 attempt 还是 retry、以及 replacement 是否完成，磁盘上可能留下 `pending`、`failed` 或 `delivering`。Worker 会把 durable state 报告为不确定，并要求调用方或 recovery code 在判断结果前重新加载。Adapter 的 delivered 或 uncertain outcome 不会仅因为状态记录返回错误就被转换成 retryable failure。

该 worker 是 deterministic single-step component，而不是自行启动的后台线程。后续具体 runtime orchestration 负责 wakeup 和重复 polling。`StateStore::reconcile_outbound_deliveries_at_startup` 为进程终止后遗留的 `delivering` record 提供平台无关 recovery：它保留 ambiguous attempt 供显式 reconciliation，而不能盲目重试。真实 provider send API 仍属于 M5，具体 adapter 可以在 provider 支持时使用幂等 key 或 delivery-status query 进一步完成 reconciliation。

## Outbound Startup Reconciliation

Outbox recovery 边界复用现有 `uncertain` status 作为显式的 reconciliation-required state。Durable `delivering` record 表示 adapter handoff 可能已经发生，但 startup 无法区分该情况与 claim replacement 已经可见、worker 尚未调用 adapter 的情况。`StateStore::reconcile_outbound_deliveries_at_startup` 在持有进程内 write lock 时重新加载 state，把所有遗留的 `delivering` record 转为带非空 recovery diagnostic 和非倒退 timestamp 的 `uncertain`，并报告全部 `uncertain` delivery id。已有 `uncertain` record 保持不变并进入同一份报告，使 durability-ambiguous write 后的重试可以重建完整 unresolved set。

Recovery report 只包含 id，不包含 adapter attempt 或 retry handoff。Pending 和可重试的 failed record 继续由普通 worker policy 管理，delivered record 不需要 recovery。Recovery 不增加 attempt count、不调用 adapter、不重试发送，也不猜测 provider outcome。

Uncertain delivery 只能通过显式 store operation 离开 reconciliation。Provider query 或 operator evidence 确认已接受时，可以调用 `StateStore::resolve_outbound_delivery_as_delivered`。只有确认未接受时才能调用 `StateStore::resolve_outbound_delivery_as_failed`；该操作要求非空 reason，并在不清除 attempt history 的情况下把 record 交回现有 bounded retry policy。正常 resolution 拒绝 `uncertain` 以外的 source state，并保持 timestamp 不倒退。仅为在 ambiguous write 后重新建立 parent-directory durability barrier，幂等 replay 可以接受已经匹配的 target state；两种 replay 都必须匹配已持久化的 resolution timestamp，failed-state replay 还必须匹配已持久化的非空 reason，并且两种 replay 都不能修改 record。

Transition 和 resolution write 必须在返回 report 或 resolved record 前完成。Pre-replace failure 保持之前 state 可见。Post-replace failure 不返回成功；后续 recovery call 或 exact resolution replay 只有在重新建立 parent-directory durability barrier 后，才能从 durable state 重建 classification 或确认 resolution。Missing state 保持不存在；重复和同进程并发 recovery 返回同一个 durable classification；同进程 matching resolution 幂等，同进程 conflicting resolution 只有一个赢家；stale public snapshot save 必须 fail closed，不能恢复 unresolved state。Cross-process `StateStore` writer coordination 在 single-daemon ownership 边界下仍不属于 M3。

本切片没有递增 runtime state schema version，因为它没有新增 serialized field 或 status value：已经能够持久化 outbound delivery 的版本也已经理解 `uncertain`。真实 provider status query 仍属于 M5 adapter 工作，自动 startup wiring 仍属于后续 concrete runtime orchestrator。

## Adapter Surface

`ImAdapter` 既 ack 经过 runtime 批准的 inbound delivery，也接收 outbox worker 提供的 normalized `OutboundDeliveryAttempt`。Attempt 带有稳定 delivery id，因此具体 adapter 可以在 provider 支持时使用幂等能力。结构化 outbound failure contract 要求 adapter 区分“确认未被接受的请求”和“结果不明确的 transport outcome”。Provider request types 和 API clients 不会穿过该边界。

真实 provider transport 细节留在具体 IM adapter 内部。Core runtime 只看到 normalized events、acknowledgement tokens 和 adapter results。

`AgentAdapter` 每次接收一个 normalized `AgentRunRequest`。该 request 从 durable `RunInputRecord` 派生，包含稳定 run id、session id、按序 user messages 和经过验证的 workspace root。Runtime 只有在拥有 pending run 且授权 exact workspace 后才构造该值；adapter 不能替换 durable input，也不能选择另一个 working directory。

Adapter 返回包含一条或多条同 session、agent-authored message 的 `AgentRunOutput`，或者返回 `AgentRunFailure`。Successful output 和 definite failure 都断言 adapter 拥有的全部 execution（包括 process tree）已经终止；只有这时 runtime 才能 complete 或 fail run 并释放其 scope。Uncertain failure 表示 adapter 无法证明 execution 已终止，因此 orchestration 必须保留 non-terminal ownership 供显式 recovery，不能释放 scope 或自动 retry。Filesystem 或其他本地 side effect 本身不会让已经终止的 execution 变为 uncertain。

空 output、user/system-authored output 和属于其他 session 的 output 会在构造 normalized output 时被拒绝。Adapter 无法产生 valid output 时，只有在仍能证明 execution 已完全终止的情况下才能报告 definite failure，否则必须报告 uncertain failure。

这个 M3 capability 有意保持 synchronous、completion-oriented。它定义 concrete orchestrator 所需的 provider-neutral handoff，但不启动进程。Process handles、streaming `AgentEvent`、stdout/stderr parsing、timeout、cancellation、backpressure 和 process-tree termination 仍属于 M4。后续加入这些机制时，必须保留这里定义的 request identity、workspace ownership 和 definite-versus-uncertain failure semantics。

## Durable Agent Run Transitions

Concrete orchestrator 在 `pending -> running` transition durable 之前，不得把 run 交给 agent。Store-level start operation 在 process-local write lock 下重新加载 state，要求存在匹配的 immutable `RunInputRecord`，执行 run transition 并持久化 replacement，之后才返回可转换为 `AgentRunRequest` 的 input。同进程两个 caller 不能同时获得同一 run 的 handoff。Start write 失败时不返回 handoff；由于 replacement 之后的错误仍可能让 `running` 可见，caller 必须 reload 或使用 startup recovery，不能推测性调用 adapter。

Successful adapter result 通过一次 store transaction 提交：每条 output message 都变为 pending `OutboundDeliveryRecord`，同时 run 在同一个 state replacement 中变为 `completed`。Store 根据 run id 和 output order 派生 stable delivery ids，不接受 adapter 选择的 id；`RunRecord::output_delivery_ids` 保存完整且有序的 ownership link。该 operation 要求 running run、exact session ownership、非空且 agent-authored 的 output、唯一 delivery ids 和不倒退的 timestamp。它不能暴露“已 completed 但没有对应 outbox work”，也不能暴露“已有 outbox work 但 run 仍是 running”。Durability-ambiguous write 之后，即使 outbox worker 已推进 delivery status，exact same completion replay 仍可确认已持久化结果；变化的 message、ordering、timestamp 或 run ownership 必须 fail closed。

Definite adapter failure 会将 running run durable 地转为 `failed`，仅在已经确认 execution termination 后释放 scope。Uncertain failure 会 durable 地转为 `interrupted`，保留 non-terminal ownership 供显式 recovery。Exact same-target replay 只用于在 ambiguous write 后重新建立 durability barrier；冲突的 terminal 或 interrupted outcome 必须 fail closed。Public snapshot save 可以携带未变化的 durable agent run，或携带由 merge 保留 durable descendant 的 stale ancestor，但不能为具有 durable input 的 run 创建 `running` 或 `completed` 状态，也不能新增或修改 output ownership。因此 caller 无法通过 `StateStore::save` 创建 handoff 或绕过 atomic completion transition；既有 failed、cancelled 和 interrupted resolution path 保持可用。这些 primitive 不执行 policy evaluation、不选择 workspace、不调用 adapter、不调度 queue wakeup，也不运行 outbox worker；这些 sequencing responsibility 仍属于后续 concrete orchestrator。

Output ownership 将 runtime state schema 从 version 7 升级为 version 8。Version 7 file 迁移时使用空 `output_delivery_ids`；version 1 到 7 不曾定义该关系，因此会拒绝非空 output link。Version 8 要求每个 run 显式包含 `output_delivery_ids`，并验证每个 linked delivery 恰好存在一次、属于 run session、包含 agent-authored message，且 creation timestamp 等于 run completion timestamp。历史 completed run 可以保留显式的空 link，而通过 store primitive 新完成的 agent run 必须有非空 output。

## Policy Foundations

M3 policy foundations 是纯内存、平台无关的 decision components。`PolicyDecision<R>` 返回 `Allowed`，或者返回带 typed reason 的 `Denied`，使 orchestration 可以记录并测试准确的 policy outcome，而不把 provider request type 或 transport behavior 嵌入 core policy。

`WorkspacePolicy` 是经过验证的 `WorkspaceRoot` exact allowlist。Workspace root 必须是 absolute、不是 filesystem root，并且不包含 parent-directory component。在 Windows 上，只接受 disk 和 UNC path（包括对应的 verbatim form）；device namespace 与 generic verbatim prefix 会被拒绝。Evaluation 会拒绝 invalid path 和未出现在 allowlist 中的 path；允许一个 parent 不会隐式允许其 descendant，也不会允许具有相同字符串 prefix 的 working directory。经过验证的 root 会保留调用方提供的 lexical `OsString` identity，而不会重写 separator、prefix kind 或 path alias。该 policy 只做 lexical validation，不访问 filesystem、不解析 symbolic link，也不判断 path 是否存在。后续 configuration 和 process-launch 边界仍负责在构造 allowlist 或启动 agent 前完成所需的 canonicalization 和 existence check。

`AccessPolicy` 对 provider-neutral `AccessPrincipal`、session scope 和 `AccessAction` 做 decision。Principal 分别保留非空 platform 和 subject namespace，避免 adapter 把不同 provider 的身份折叠到一起。Policy 使用 exact principal/scope allowlist，默认拒绝所有 request；`AdministerBridge` 只允许 administrator set 中的 principal，并且 administrator set 必须是普通 allowed principals 的子集。`InvokeAgent` 是独立的非管理 action。Denial 会区分 unknown principal、unknown scope 和非 administrator 发起管理 action。

本切片不会在 persistent config 或 runtime state 中新增 policy field，不会从 message 提取 provider identity，不会生成 policy log，也不会把 policy evaluation 接入 `RuntimeOrchestrator`。这些 integration point 属于后续 concrete orchestration 和首个 provider milestone，因此本切片不会修改 runtime state schema。

## 验收说明

Inbound intake 切片完成时，应满足：重复 inbound deliveries 在 acknowledgement 之前被记录或识别，持久化失败不会 ack，acknowledgement 失败在 durability 之后报告，同进程同路径 store handles 不会丢失已 ack 的 inbound records，stale saves 会保留已有 inbound ledger entries，并且 architecture/roadmap 文档说明真实 transports 和 cross-process orchestration 仍不在当前范围内。

Per-scope message queue 与 exclusion 切片完成时，应满足：绑定 session 的 inbound message 在 acknowledgement 前原子持久化 ledger 和 queue entry；duplicate event 在 claim 前后都不会重复入队；未知 session 和损坏的 queue、run-input 或 ledger 关系会 fail closed；debounce 按 scope 隔离；full batch 无需等待即可 ready；观察结果确定且不消费消息；durable claim 会创建一个 pending run 和可恢复 input，并只移除 bounded queue prefix；pending、running 和 interrupted run 会阻止该 session 的另一次 claim，而 terminal history 不会；同进程并发 claim 最多返回一个 handoff；pre-replace failure 保留 queued work；post-replace ambiguity 不返回 handoff但保留可恢复 ownership；stale save 不能擦除 claim 或恢复已领取消息；public snapshot 不能伪造 run input；version 4 迁移为空 queue 的 version 5；version 5 迁移为空 run inputs 的 version 6。Run startup recovery 使用下方独立验收边界。

Run startup recovery 切片完成时，应满足：具有 durable input 的 pending run 保持 pending，并且报告不包含 execution handoff；running 和缺少 input 的 pending run 转为 non-terminal interrupted owner；interrupted ownership 在显式解决为 failed 或 cancelled 前继续阻止该 scope；已有 failed run 保持可检查且不会自动 retry；completed 和 cancelled history 不进入报告；重复和并发调用返回相同 durable classification；时钟回拨不会使 run timestamp 倒退；pre-replace failure 保留旧状态；post-replace failure 不返回报告，但后续调用可以重建；stale save 不能用 running 覆盖 interrupted；缺失 state 不会被创建；version 6 迁移到 version 7，同时历史版本拒绝 interrupted run。

Outbound outbox 切片完成时，应满足：outbound records 验证 session/message ownership，状态转换拒绝无效 delivery shape 和倒退时间戳，version 3 state files 会迁移为空 outbox，version 4 state files 要求显式 outbound deliveries，enqueue 只接受已知 session 上的 pending records 且只有在 durable persistence 后才返回，重复 enqueue 是幂等的，public snapshot save 不能引入 non-pending delivery，同进程同路径 store handles 不会丢失 queued outbound records，stale saves 会保留已经 queued 的 deliveries，或在依赖缺失时 fail closed 且不替换现有文件。

Outbox consumption primitive 切片完成时，应满足：claim 会在返回前把恰好一个 pending 或 failed delivery 转为 `delivering` 并持久化，failed deliveries 可以再次 claim 用于 retry，没有可 claim record 时不返回 delivery，同进程同路径 store handles 不能重复 claim 同一个 delivery，claim 持久化失败会返回错误而不是 handoff，并且 delivered 或 failed outcome 会在返回更新后的 record 前 durable。

Outbox worker 切片完成时，应满足：只有已到期 delivery 才会在 durable claim 后交给 adapter；handoff 包含 normalized scope 和稳定 idempotency key；成功、可重试失败和 uncertain send 都会在返回 outcome 前 durable；只有确认未被接受的 failure 才进入 bounded exponential backoff；uncertain outcome 保持可检查但不会再次 claim；已耗尽 failure 保持可检查但不会再次 claim；deferred failure 不会阻塞后续 eligible record；claim 持久化失败绝不会触发 adapter 调用；outcome 持久化错误会被视为 durability 不确定，而不会触发不安全的立即重试。

Outbound startup reconciliation 切片完成时，应满足：startup 可见的每个 `delivering` record 都会转为 `uncertain`，不产生新的 adapter handoff 或 attempt increment；报告全部已有和 recovery 产生的 uncertain delivery id；pending、failed 和 delivered record 保持不变；只有显式确认已接受或确认未接受的 resolution 才能离开 `uncertain`；确认未接受后，在不丢失 attempt history 的情况下回到 bounded retry policy；timestamp 不倒退；missing state 不会被创建；pre-replace 和 post-replace failure 不会提前返回成功；exact same-target replay 会确认 durability 并拒绝变化的 timestamp 或 reason；重复和同进程并发调用重建一个 durable result；stale save 不能恢复 unresolved state；该切片保持当时的 runtime state schema version 7 不变。

Policy foundation 切片完成时，应满足：decision typed 且 deterministic；空的 workspace 或 access policy 默认拒绝；workspace root 拒绝 relative path、包含 parent component 的 path、filesystem root，以及 Windows device namespace 和 generic verbatim prefix；workspace authorization 保留 lexical path identity，并要求 exact allowlist match，而不是重写 separator/prefix、授权 descendant 或做字符串 prefix match；principal 保留 platform 和 subject namespace；普通 access 同时要求 principal 与 session scope 已进入 allowlist；管理 access 还要求 administrator 已经属于 ordinary allowed principals；malformed administrator configuration fail closed；并且本切片不新增 persistent config、runtime-state schema、provider identity extraction、filesystem lookup 或 orchestration side effect。

Agent adapter capability 切片完成时，应满足：request 保留 durable run 和 session identity，在不让 persistence 或 provider type 穿过边界的情况下暴露按序 user messages，并且只携带经过验证的 exact workspace root；output 至少包含一条属于 request session 的 agent-authored message；successful output 和 definite failure 断言整个 execution tree 已终止，uncertain failure 表示无法证明终止；invalid output 也遵循同一个基于 termination 的分类；trait 自身不执行 persistence 或 policy decision；本切片不新增 process execution、event streaming、timeout、cancellation、concrete adapter 或 runtime-state schema 变更。

Durable agent-run transition 切片完成时，应满足：只有 pending run 已 durable 地变为 running 后才返回 adapter-eligible input；缺少 input、非 pending state、同进程竞争和 persistence failure 都不返回 handoff；successful output 必须非空且 agent-authored，获得 runtime-owned stable delivery ids，并与 pending outbox records 及 run completion 原子关联；definite failure 转为 terminal failed，uncertain failure 转为 non-terminal interrupted；timestamp 不倒退；exact same-target replay 可以确认 ambiguous replacement，冲突 replay 必须 fail closed；stale public save 不能撤销 transition、擦除已提交 output 或伪造 output ownership；version 7 以空 output link 迁移到 version 8，历史版本拒绝新 linkage；并且本切片不新增 policy sequencing、adapter call、queue loop 或 process execution。
