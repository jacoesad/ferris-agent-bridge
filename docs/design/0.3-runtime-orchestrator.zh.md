# 0.3 Runtime Orchestrator 设计

## 目标

Milestone 3 在 runtime foundations 之上加入 durable orchestration，但暂不引入真实 IM provider 或真实 agent CLI adapter。Runtime 应拥有跨平台的 ordering、duplicate handling、acknowledgement preconditions、queueing、recovery 和 policy decisions。具体 provider 和 agent 仍留在 adapter 边界之后。

本文档记录 M3 在多个聚焦 PR 中逐步引入的设计约束。早期 `0.x` 阶段它不是兼容性承诺。

## 边界

包含：

- 用于重复投递处理的 durable inbound event ledger。
- Inbound events 的 store-level ack-after-persist persistence primitive。
- 初始 runtime intake 边界：只有 durable record 或 duplicate detection 成功后，才 ack inbound delivery。
- Runtime 拥有的 event identity 规则，用于 provider/scope-safe duplicate detection。
- 同路径 `StateStore` read-modify-write helpers 的进程内串行化。
- Durable outbound outbox records，以及 send 前必须 enqueue 的 store-level persistence primitive。
- 后续 M3 queueing、outbox consumption、startup recovery 和 policy skeletons。

不包含：

- 真实 Lark / 飞书、Slack、Discord 或其他 provider transport。
- 真实 agent process execution。
- daemon ownership 之外的 cross-process queueing 或 cross-process `StateStore` writer coordination。
- 完整 platform SDK abstraction。

## Inbound Delivery Intake

有些 inbound transport 需要显式 delivery acknowledgement。Runtime 必须先持久化或去重归一化后的 event，再让 adapter ack 平台 delivery。

初始形态是：

```text
InboundDelivery -> RuntimeOrchestrator::accept_inbound_delivery -> ImAdapter::acknowledge_inbound_delivery
```

`RuntimeOrchestrator::accept_inbound_delivery` 通过 `StateStore::persist_inbound_event` 记录 event。当持久化返回 `Recorded` 或 `Duplicate` 时，orchestrator 创建 acknowledgement 对象并交给 adapter。如果持久化失败，不会调用 adapter，平台 delivery 保持未 ack，让 transport 按 provider 语义重试。如果 adapter acknowledgement 失败，错误会在 event 已经 durable 之后返回。

Acknowledgement value 由 crate 创建，而不是由外部调用方创建，因此 adapter 不能伪造一个跳过 runtime persistence 的 acknowledgement。

## Event Identity

`EventId` 是 duplicate-detection key。Provider delivery identifier 通常只在某个 provider、app、tenant、chat、channel 或 receive stream 内唯一，所以 adapter 在把 event 交给 runtime 前，必须给 inbound delivery id 加命名空间。

`EventId::for_platform_delivery(platform, scope, delivery_id)` 提供默认 canonical shape。构造完成后，runtime 把这个 id 当作 opaque value。

## StateStore Safety

M3 引入的 read-modify-write helpers 可能通过 cloned 或独立构造的 `StateStore` handles 触发。因此同一路径 handles 共享一把进程内写锁，并且 store 会先规范化 path aliases，再选择 lock key 和 I/O path。这样可以避免两个进程内 inbound deliveries 都被 ack，但其中一个 ledger entry 被 last-writer-wins replacement 覆盖丢失。

`StateStore::save` 在保存较旧的内存快照时，会保留磁盘上已经存在的 inbound ledger entries。如果现有文件不能读取、解码或验证，它会 fail closed，因为覆盖 invalid state 可能擦掉已经 ack 的 delivery。

同样的 stale-save 保护也适用于 outbound outbox records。较旧的内存快照不能擦掉另一个同路径 `StateStore` handle 已经 durably queued 的 delivery。如果同一个 outbound delivery id 对应不同 payload 或状态，或者保留已有 outbound delivery 需要 candidate state 中不存在的 session，保存会 fail closed，而不是采用 last-writer-wins 或替换现有文件。

这把锁不是跨进程队列。当前 daemon lifecycle 保证一个被拥有的 runtime 进程；如果以后项目加入更多 writer，后续 orchestrator 工作应显式设计 cross-process runtime coordination。

## Outbound Outbox

Outbound replies 必须先记录到 durable state，后续 adapter 才能发送。`OutboundDeliveryRecord` 保存 delivery id、目标 session、agent/system-authored message、delivery status、timestamps、attempt count、可选 delivered time 和可选 last error。

初始 outbox 状态包括：

- `pending`：已经排队，尚未尝试发送。
- `delivering`：已被领取用于一次发送尝试。
- `delivered`：已经成功交给未来的 IM adapter。
- `failed`：尝试发送失败，并带有非空 last error。

`StateStore::enqueue_outbound_delivery` 使用与 inbound delivery persistence 相同的 path-local write lock 执行 read-modify-write。它只接受已知 session 上的 pending record，只有在 delivery 已经 durable 后才返回 `Queued`，对完全相同的已排队记录返回 `Duplicate`，对冲突 id 或持久化失败 fail closed。

这个切片还不实现 outbox worker、retry scheduler、backoff policy 或具体 IM send API。这些属于后续 M3 queueing/orchestration 和 M5 provider integration 工作。

## Adapter Surface

初始 `ImAdapter` capability 故意很窄：只 ack 经过 runtime 批准的 inbound delivery。Outbound delivery records 现在已经是 durable runtime state，但具体 IM send capability 和 agent 侧能力会由后续 M3/M4 PR 引入。

真实 provider transport 细节留在具体 IM adapter 内部。Core runtime 只看到 normalized events、acknowledgement tokens 和 adapter results。

## 后续 M3 工作

Ack intake boundary 只是 M3 的一个切片。该里程碑仍需要：

- Per-scope message queue，包含 debounce 和 batching。
- Outbox consumption、retry，以及具体 outbound adapter handoff。
- Active runs 的 scope-level mutual exclusion。
- Pending、running 和 failed runs 的 startup recovery。
- Workspace 和 access policy skeletons。
- 串联 storage、queues、policies 和 adapter boundaries 的具体 runtime orchestration。

## 验收说明

Inbound intake 切片完成时，应满足：重复 inbound deliveries 在 acknowledgement 之前被记录或识别，持久化失败不会 ack，acknowledgement 失败在 durability 之后报告，同进程同路径 store handles 不会丢失已 ack 的 inbound records，stale saves 会保留已有 inbound ledger entries，并且 architecture/roadmap 文档说明真实 transports 和 cross-process orchestration 仍不在当前范围内。

Outbound outbox 切片完成时，应满足：outbound records 验证 session/message ownership，状态转换拒绝无效 delivery shape 和倒退时间戳，version 3 state files 会迁移为空 outbox，version 4 state files 要求显式 outbound deliveries，enqueue 只接受已知 session 上的 pending records 且只有在 durable persistence 后才返回，重复 enqueue 是幂等的，同进程同路径 store handles 不会丢失 queued outbound records，stale saves 会保留已经 queued 的 deliveries，或在依赖缺失时 fail closed 且不替换现有文件。
