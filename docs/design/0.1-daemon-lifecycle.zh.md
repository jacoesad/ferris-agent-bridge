# 0.1 Daemon 生命周期设计

本文记录 `0.1.0` 的 daemon lifecycle 实现，以及后续应该如何演进。它的范围有意小于完整 bridge runtime。

## 范围

第一个 daemon 里程碑提供本地生命周期命令：

- `ferris-agent-bridge start`
- `ferris-agent-bridge start --foreground`
- `ferris-agent-bridge status`
- `ferris-agent-bridge stop`

本里程碑不包含 IM adapter、Lark / Feishu channel 集成、OpenAPI 调用、真实 agent 执行、会话连续性或流式回复。

目标是在任何远程 chat event 能触发本地任务之前，先证明项目可以安全管理一个本地后台进程。

## 支持策略

`0.1.0` 以 macOS 作为第一个经过验证的支持目标。

本里程碑的 CI 策略：

- macOS 是 required gate，运行 formatting、Clippy、locked build 和 locked tests。
- Linux 和 Windows 是 build-only compatibility gates。它们运行 locked build，用于提前发现明显编译回归，但 daemon lifecycle 行为暂不作为支持契约。

## 当前模型

CLI 层负责解析命令，并把 daemon 行为交给 daemon 模块。Daemon 模块负责本地 runtime 路径、状态检查、lock 处理、进程启动、stop 请求和清理。

当前生命周期状态是：

```text
Running
RunningUnverified
Stopped
Stale
Unowned
```

这些是 daemon 进程状态，不是未来的 runtime 或 run 状态机。`0.2.0` 规划中的 runtime 状态机应建模 sessions、queued events、run lifecycle、recovery、acknowledgements 和 outbound delivery。

## Runtime 目录

默认 runtime 目录是：

```text
~/.ferris-agent-bridge
```

可以通过环境变量覆盖：

```text
FERRIS_AGENT_BRIDGE_HOME
```

当前文件包括：

- `daemon.lock`：启动与归属 lock。
- `daemon.state`：记录 daemon metadata。
- `daemon.stop`：包含 daemon token 的 stop 请求。
- `daemon.log`：本地 daemon 日志输出。
- `daemon.lifecycle.lock`：临时 lifecycle operation lock。

`daemon.lock` 使用 create-new 语义获取，这样并发 start 不可能同时成为 owner。Daemon 进程 ready 后，会用原子替换写入归属 metadata。无效 lock 内容会被视为 unsafe，不会被自动删除。

`daemon.state` 记录 process id、ownership token、executable path、启动时间戳和运行模式。它使用原子替换写入，避免 `status` 和 `stop` 读到半写 state。Ownership token 用于避免 `stop` 只信任 PID。

在 Unix 和 macOS 上，runtime 目录限制为 `0700`，daemon 文件限制为 `0600`。Windows-specific ACL hardening 应在后续 OS service adapter 层引入时最终确定。

## 命令行为

`start` 会检查当前状态、移除 stale state、获取 lifecycle lock、写入 `starting` lock、启动内部 daemon 进程，在支持的平台上让后台子进程脱离调用方 session，释放 lifecycle lock，并短暂等待包含预期 child PID 和 ownership token 的 ready record。内部 daemon entrypoint 必须获取 lifecycle lock，并确认当前 lock 是同一个 token 和 starter PID 对应的 `starting` lock，然后才能替换 `daemon.lock` 或写入 `daemon.state`。启动 ready 不应依赖外部进程 command-line inspection。

`start --foreground` 使用同样的 state 和 stop-request 路径，但把 daemon loop 留在前台，方便开发和调试。

`status` 会读取记录的 state，并校验进程是否仍然存活。如果 command-line identity 证据可用，它会校验 daemon token 和内部 entrypoint。如果 command-line identity 证据不可用，status 可以报告 daemon 正在运行但身份未验证。

如果 runtime 目录不存在，`stop` 会直接返回 not running，且不会创建 runtime 目录。如果 lifecycle 文件已存在，它会获取 lifecycle lock、写入带 token 的 stop request，并等待进程优雅退出。Daemon 进程观察到 stop request 后只负责退出；lifecycle 文件清理由 `stop` 命令在持有 lifecycle lock 时完成。只有归属已验证时，才允许超时后退回到进程终止。发送 force-stop 信号前，必须重新读取 state 并重新检查进程身份，避免 stale PID 在 PID reuse 后被误杀。它会拒绝 force-stop 无法确认归属或未验证的进程。

Cleanup 是 ownership-conditional 的。Stop 或 failed-start 路径只应删除 metadata 仍然匹配当前 daemon record 或 token 的 runtime 文件。Daemon lifecycle 文件的 mutation 会通过 `daemon.lifecycle.lock` 串行化，因此 ownership 匹配和删除处在同一个 lifecycle 临界区内。这样可以避免旧 cleanup 路径误删已经由新 daemon start 写入的文件，同时避免引入更细粒度的文件锁协议。

Lifecycle lock 的等待超时应长于预期最慢的 lifecycle operation。目前的取值为 graceful stop 加 verified force-stop fallback 留出了余量。

重复 `start` 不应创建多个 active daemon 进程。

## 安全规则

Daemon foundation 后续演进时应保持这些规则：

- 不只信任 PID。
- 无法读取或格式无效的 state 应视为 stale 或 unsafe，而不是 running。
- 进程归属无法验证时，拒绝 stop。
- 缺少 command-line identity 证据时，应视为 unverified，而不是可信。
- 内部 daemon entrypoint 写入 runtime state 前，必须通过 startup lock 校验。
- 启动 ready 不应依赖外部 command-line inspection。
- force-stop daemon 前，必须重新检查 state 和进程身份。
- 启动 lock 获取必须保持原子性。
- lock 与 state metadata 更新必须保持原子性。
- 会修改 daemon lifecycle 文件的 lifecycle operation 应串行化。
- runtime 目录和 daemon 文件应只对当前用户可访问。
- cleanup 应保持 ownership-conditional，避免 stale cleanup 删除新 daemon 的文件。
- 清理逻辑应 best-effort，但行为确定。
- daemon lifecycle 应保持独立，不依赖任何 IM 平台或 agent CLI。

## 与 OS Service 的关系

当前实现是直接启动一个子进程。这足够支撑第一个本地生命周期里程碑，但它还不是完整 OS service manager。

后续演进应拆开两个概念：

- `run` 或内部 daemon entrypoint：运行 bridge runtime 进程。
- `start` / `stop` / `status`：在启用 OS service 支持时，管理平台服务注册。

平台 service 层应隐藏在一个小的 adapter 边界之后。可能的实现包括：

- macOS：launchd LaunchAgent。
- Linux：systemd user unit。
- Windows：Task Scheduler。

Service 层应负责 service registration files、OS-specific status parsing、restart behavior、autostart behavior 和 stdout/stderr log routing。Runtime 进程仍应负责 bridge state、locks、sessions 和 adapter orchestration。

## 未来多 Agent 锁设计

`daemon.lock` 是进程级 daemon lock。它不应该演变成所有未来 agent、profile 或 chat app 的业务互斥锁。

长期更推荐的模型是：一个本地 daemon 进程在内部协调多个 profiles 和 agent kinds。业务层互斥应使用更细粒度的锁，例如：

```text
registry/locks/profile/<profile>.lock
registry/locks/app/<app_id>.lock
registry/locks/agent/<agent_id>.lock
```

Profile lock 防止同一个 profile 同时运行冲突任务。App lock 防止两个 runtime 竞争同一个 chat platform app 或 event connection。Agent lock 只有在某个本地 agent 资源确实不能安全并发运行时再引入。

这样可以保持 OS service 管理简单，同时允许一个 daemon 后面挂多个已配置 agent。

## 演进计划

`0.1.0` 应完成本地 daemon lifecycle：

- 保留当前本地 daemon 命令。
- 加强 duplicate starts、stale state 和 stop behavior 的集成测试。
- 以 macOS 为第一个目标，同时避免不必要的平台耦合。

`0.2.0` 应加入 runtime foundation：

- Session 与 queue state。
- Run state machine。
- Inbound event ledger。
- Ack-after-persist contract。
- Outbound outbox。
- Startup recovery。

后续 service-management 工作可以添加 OS service adapters，而不改变 runtime 状态机或 IM adapter contracts。
