# 0.2 Runtime Foundations 设计

## 目标

0.2 foundation line 建立平台无关的 runtime 基础层。已发布的 `0.2.0` 版本引入本地配置、持久状态、会话身份、通用 message / event 模型、结构化日志、脱敏和错误策略，供后续 runtime orchestrator 和 adapter 继续构建。`0.2.0` 之后在 `main` 上继续进行 foundation hardening，并随下一个版本发布。

这个里程碑不启动真实 agent run，不消费聊天事件，不定义 adapter trait，也不实现队列。这些属于后续里程碑。

## 边界

包含：

- 带版本号的本地配置，包含 `secret: Option<SecretInput>` 占位。
- 具备原子写语义的 runtime state 存储，并在 Unix/macOS 上为新建 JSON 文件使用私有权限。
- 基于 platform 和 scope 的稳定 session identity。
- 具备本地 pending/running/terminal 状态迁移的 run records（`0.2.0` 之后的 hardening）。
- 平台无关的 `Message` 和 `Event` 领域模型。
- 携带 session、event 和未来 run 上下文的结构化日志。
- 对结构化字段和行内值进行 secret 脱敏。
- 将 runtime 错误分类为 fatal、recoverable、user-visible。

`0.2.0` release 未包含：

- 真实 agent 执行、scheduler 驱动的 run orchestration，以及 run recovery records。
- Inbound event ledger 和 outbound outbox。
- 消息队列、debounce 或 batching。
- `ImAdapter` 和 `AgentAdapter` trait。
- Workspace / access policy 评估。
- Lark / 飞书、Slack、Discord 或具体 agent 的实现细节。

## Config

`BridgeConfig` 从一开始就带 `version`，这样后续版本可以迁移旧配置，而不是破坏用户本地环境。第一版配置刻意保持很小：

- `version`：配置格式版本。
- `profile`：用户选择的本地 profile 名。
- `runtime`：相对 profile/runtime 目录的 state 和 log 文件名。
- `secret`：给未来 keystore-backed 值预留的占位。

`SecretInput` 只描述环境变量、文件这类引用来源，不把明文 secret 写入配置。加密本地 keystore 延后到聊天平台里程碑中处理，因为那时才会引入真实平台凭证。

## State

`StateStore` 使用 JSON 持久化 runtime 拥有的状态，并通过同目录临时文件加 `rename` 写入。这样 daemon 和未来 runtime 会共用一条可靠的持久化路径，避免进程崩溃后留下半写入状态文件。

持久化 state 文件拥有 schema version。`StateStore` 在 load/save 时负责包装和解包文件格式，而 `RuntimeState` 只保留内存中的 runtime model。

在 Unix/macOS 上，新建的 runtime JSON 文件和缺失的父目录会在创建时使用私有权限。Windows 上，本里程碑先保留同一套 atomic-write API 和兼容性覆盖，但暂不显式 harden ACL；在这些 helper 存储明文平台凭证前，应补上 Windows-specific ACL 隐私处理。

已发布的 `0.2.0` state 包含 sessions。`0.2.0` 之后的 foundation hardening 增加 run records，并开始加入 M3 的第一块 inbound event ledger state。Outbound outboxes 和 recovery records 仍延后到后续 `0.3.0` runtime-orchestrator 工作中处理。

## Sessions

Session identity 对 `(platform, scope)` 保持稳定。`scope` 故意保持通用：对 Lark 来说以后可能是 chat id 或 thread id；对其他平台可能是 channel、conversation 或 direct-message scope。

Session 层不能知道 provider SDK 类型。平台 adapter 需要先把 provider 标识归一化成通用 `SessionScope`，再交给 runtime。

Session 时间戳属于 continuity model 的一部分：`created_at_unix` 记录 runtime 第一次看到该 session 的时间，`updated_at_unix` 不能早于 `created_at_unix`，也不能在 refresh 时倒退。

## Messages and Events

`Message` 表示桥接层的内容、作者和可选 session 关联。`Event` 表示标准化后的 runtime 输入/输出事实，比如收到消息、session 开始、runtime notice。

这些是平台无关的领域模型。Provider 请求 payload、原始事件 envelope、receive id、card action 和 media resource 都应留在平台 adapter 内部。

## Logging and Redaction

结构化日志包含 level、message 和 context fields。context 支持：

- `session_id`
- `event_id`
- `run_id`，为后续里程碑预留
- adapter/runtime 诊断用的自由字段

token、secret、password、authorization 这类字段在输出日志前会被脱敏。这个策略同时适用于结构化字段，以及 message 里的 `key=value` 或 `key:` 行内模式。

## Error Policy

Runtime 错误按行为分类：

- `fatal`：进程无法安全继续。
- `recoverable`：当前操作失败，但 runtime 可以继续运行。
- `user_visible`：失败应以结构化错误返回给调用方或聊天用户。

后续 adapter 必须把 provider 和 agent 失败映射到这套策略，而不是各自发明一套失败行为。

## 验收说明

这个里程碑完成时，应满足：config 和 state 可以通过共享 atomic write 路径保存/加载，Unix/macOS 上新建的 runtime JSON 文件使用私有权限，session 拥有稳定标识，message/event 模型能通过 JSON 往返，日志输出会脱敏 secret-like 数据，并且已有 daemon lifecycle 测试继续通过。显式 Windows ACL hardening 延后处理。
